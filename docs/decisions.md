## 决策 1：P0 默认慢权重模型

| 维度 | 选项 A | 选项 B | 选项 C（当前倾向） |
|---|---|---|---|
| 模型 | `sentence-transformers/all-MiniLM-L6-v2` | `BAAI/bge-small-zh-v1.5` | 无默认值 |
| 维度 | 384 | 512 | 由用户模型决定 |
| 语言 | 英文为主 | 中文优化 | 不限 |
| 优点 | 轻量、生态最广、句子级训练 | 中文场景效果好 | 灵活，不绑定领域 |
| 缺点 | 中文一般 | 英文非最优 | 用户需要多一步配置 |

**讨论**：

P0 的核心是验证原型可塑网络机制，慢权重模型本身不是变量。强制用户传入 `base_model` 可以避免早期被语言/领域绑架，也让测试更明确。P1 再根据实际用户反馈提供推荐默认值。

**阻塞点**：确认目标用户语言场景；或者 P1 提供 2-3 个推荐模型列表。

## 决策 2：Rust 阶段推理引擎

| 维度 | 选项 A（当前倾向） | 选项 B | 选项 C |
|---|---|---|---|
| 引擎 | `candle` | `ort`（ONNX Runtime） | `tch-rs` |
| 依赖 | 纯 Rust | ONNX Runtime 动态库 | libtorch |
| 产物大小 | 小 | 中 | 大 |
| 模型支持 | 中（增长中） | 广 | 最广 |
| 生态成熟度 | 较新 | 成熟 | 成熟 |

**讨论**：

如果目标是"一个干净的 Rust 推理后端"，`candle` 最理想。但具体能否支持 P2 选定的慢权重模型（如 BGE 系列）需要验证。P2 启动前做一次 PoC：用 candle 加载目标模型并跑出正确 embedding，再决定。

**阻塞点/决策时机**：P2 开始前评估 candle 对目标模型的支持度。

## 决策 3：持久化格式

| 维度 | 选项 A | 选项 B | 选项 C（当前倾向） |
|---|---|---|---|
| 格式 | `.pt`（PyTorch） | `.npy`（NumPy） | `.safetensors` |
| Python 读写 | 极方便 | 方便 | 需 `safetensors` 库 |
| Rust 读写 | 需自定义解析 | 方便 | 有官方 crate |
| 安全性 |  pickle 风险 | 安全 | 安全 |
| 元数据支持 | 完整 dict | 仅矩阵 | 需额外 JSON |

**讨论**：

P0 用 `.pt` 最方便（torch.save/torch.load）。P2 若切 Rust，`.safetensors` 是更跨语言、更安全的选择，但元数据（step、strength 等）需要额外 JSON 文件配合。P1 末段再做最终决定。

**阻塞点/决策时机**：P2 Rust 侧加载库的复杂度评估后决定。

## 决策 4：Python/Rust 边界

| 维度 | 选项 A（当前倾向） | 选项 B | 选项 C |
|---|---|---|---|
| 架构 | 推理 Rust + 训练 Python | 全 Rust，Python 薄封装 | 全 Python，Rust 可选后端 |
| 性能收益 | 推理路径最大 | 全局最大 | 最小 |
| 开发成本 | 中（PyO3 桥接） | 高 | 低 |
| 维护成本 | 中 | 低（单一语言） | 低 |

**讨论**：

P0-P1 全 Python。P2 若性能瓶颈在 `encode()`，则优先把推理管道 Rust 化；若瓶颈在训练，则把训练也 Rust 化。先做 profiling 再决定。

**阻塞点/决策时机**：P2 明确性能瓶颈位置。

## 决策 6：预测模块具体形态

**阻塞点/决策时机**：P1 验证后再评估。

## 决策 7：自适应基线算法

| 维度 | 选项 A（当前倾向） | 选项 B | 选项 C |
|---|---|---|---|
| 算法 | 滑动窗口均值 ± 2σ | EMA 指数移动平均 | 固定阈值 |
| 响应速度 | 中等 | 快 | 即时 |
| 对异常值敏感 | 中 | 低 | 无 |
| 实现复杂度 | 中 | 低 | 最低 |

**讨论**：

surprise_score 的基线需要足够稳定，又不能永远不变。滑动窗口均值能反映近期话题分布，2σ 作为异常判定较直观。EMA 更轻量，适合高频场景。P1 实现两种算法做 A/B 对比。

**阻塞点/决策时机**：验证阶段根据数据特性选择。

## 如何参与决策

每个决策都需要在对应阶段启动前完成。决策流程建议：

1. 收集使用场景/数据证据；
2. 列出候选方案与明确 trade-off；
3. 做最小 PoC（如需要）；
4. 在 ADR（Architecture Decision Record）中记录最终选择。
